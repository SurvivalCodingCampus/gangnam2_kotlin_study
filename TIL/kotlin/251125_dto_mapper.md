# DTO
Spring에서도 DTO로 요청 값을 받고 응답을 보낼 때 DTO로 변환해서 보내줬다.  
DTO 썼던 이유는 `Entity`는 DB 데이터의 직접적인 연관있는 객체기 때문에 DTO로 변환하거나, 데이터를 조회할 때 응답 값에 불필요한 데이터를 안내려주기 위해 사용하곤 했었다.

큰 이유는 비슷한 것 같다.
- 데이터 소스를 모델 클래스로 변환하는 과정에서 순수하게 클래스에 담기 위한 **중간 전달 객체**
- 잘못된 데이터 소스(Json)를 받더라도 안 터지게 하려는 클라이언트 개발자의 방어 수단

## DTO가 필요한 이유
- Model class는 non-nullable값만 가지고 있도록 한다.(단, 무족건은 아니다. 상황에 맞게 변경하면 된다.)
- Json 데이터는 null 값을 포함할 수 있다.
- Map -> Model class 변환 시 null 값 등의 예외를 사전에 걸러내기 용이하다.
- 불완전한 데이터가 포함될 것 같으면 DTO를 도입

# Mapper
자바에서는 `static method`로 **dto to entity** or **entity to dto**로 변환하는 것을 작성했었다.

- 순수한 데이터 소스(DTO)르 원하는 모델 클래스로 변환하기 위한 로직은 별도의 mapper를 통해 변환한다.

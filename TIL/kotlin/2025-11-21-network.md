# 네트워크 통신

## 용어

### URI (Uniform Resource Identifier)

- URI는 “통합 자원 식별자”로, 인터넷 상의 어떤 리소스를 **식별**하기 위한 문자열 전체를 통칭한다.
- URI 안에는 “위치로 식별하는 방식(URL)”과 “이름으로 식별하는 방식(URN)”이 모두 포함될 수 있다.
- 예시:
    - `https://www.example.com/index.html` → 이건 위치를 나타내니 URL이면서 동시에 URI이다.
    - `urn:isbn:0451450523` → 이름을 나타내니 URN이면서 동시에 URI이다.

정리하면, “인터넷에서 어떤 걸 하나 가리키는 문자열” 정도의 가장 넓은 개념이 URI다.

### URL (Uniform Resource Locator)

- URL은 “통합 자원 **위치** 지정자”로, 말 그대로 리소스가 **어디에 있는지(위치)** 를 나타내는 URI의 한 종류다.
- 보통 우리가 말하는 “웹 주소”가 대부분 URL이다.
- 일반적인 문법 형식은 다음과 같다.
    - scheme: `http`, `https`, `ftp` 등 프로토콜
    - host: 도메인 또는 IP (`www.example.com` 등)
    - port: `80`, `443` 같은 포트 번호 (생략 시 기본값)
    - path: 서버 안의 리소스 경로 (`/search`, `/images/a.png` 등)
    - query: `?q=hello&lang=ko` 같은 추가 파라미터
    - fragment: `#section1` 처럼 문서 내 특정 위치 표시 (서버로는 안 감)

- 예시
    - `https://www.google.com:443/search?q=hello#top`
    - `ftp://user:pass@ftp.example.com/files/readme.txt`

핵심은 “URL은 리소스의 **주소(위치)** 를 알려주는 URI”라는 점입니다.

### URN (Uniform Resource Name)

- URN은 “통합 자원 **이름**”으로, 리소스의 **이름 자체**로 식별하는 URI의 한 종류입니다.
- 위치 정보는 없고, 위치가 바뀌어도 변하지 않는 **영속적인 이름**을 목표로 합니다.
- 형식은 보통 다음과 같습니다.


- `urn` : URN 스킴이라는 표시
- NID(Namespace Identifier): 네임스페이스 이름 (예: `isbn`, `uuid`)
- NSS(Namespace Specific String): 그 네임스페이스 내에서의 고유 문자열

- 예시
    - `urn:isbn:0451450523` → ISBN이 `0451450523`인 책
    - `urn:uuid:6ba7b810-9dad-11d1-80b4-00c04fd430c8` → 특정 UUID 리소스

실제 웹 개발에서는 URN은 상대적으로 잘 안 쓰이고, URL 중심으로 사용되는 경우가 많습니다.

### HTTP 개념

HTTP는 웹 브라우저(클라이언트)와 웹 서버가 **데이터를 주고받기 위해 약속해 둔 규칙(프로토콜)** 입니다.

- HTTP(HyperText Transfer Protocol)는 HTML 문서, 이미지, JSON 같은 웹 리소스를 요청·응답하는 **애플리케이션 계층 프로토콜**이다.

### HTTPS

HTTPS는 HTTP에 **암호화(SSL/TLS)를 얹어서 더 안전하게 만든 프로토콜** 입니다.

- HTTPS(HyperText Transfer Protocol Secure)는 브라우저와 서버 사이의 HTTP 통신을 SSL/TLS로 암호화해서 도청·변조·위조를 막는 보안 버전입니다.

### TCP (Transmission Control Protocol)

- 연결형 프로토콜: 통신 시작 전에 3-way handshake로 논리적인 연결을 먼저 맺고, 끝날 때 4-way handshake로 끊는다.
- 신뢰성 보장: 패킷이 빠지거나 순서가 바뀌면 재전송하고, 흐름 제어·혼잡 제어를 해서 데이터가 손상 없이, 순서대로 도착하도록 보장한다.
- 그래서 UDP보다 느리고 오버헤드가 크지만, 파일 다운로드, 웹 페이지 조회(HTTP/HTTPS), 이메일, SSH처럼 **정확성이 중요한 통신**에 사용된다.

### UDP (User Datagram Protocol)

- 비연결형 프로토콜: 연결 설정 없이 바로 패킷(데이터그램)을 던지고, 받았는지 안 받았는지 확인하는 과정이 없다.
- 비신뢰성: 순서 보장, 재전송, 흐름 제어, 혼잡 제어가 없어서 패킷이 유실되거나 순서가 바뀔 수 있다.
- 대신 구조가 단순하고 헤더가 작아서 TCP보다 빠르고 부하가 적어, 온라인 게임, 실시간 스트리밍, VoIP처럼 **몇 개 프레임이 날아가도 상관없고 끊기지 않는 게 더 중요한 통신**에 자주 쓰인다.

TCP vs UDP

| 구분    | TCP                | UDP                  |
|-------|--------------------|----------------------|
| 연결 방식 | 연결형(핸드셰이크 필요)      | 비연결형(바로 전송)          |
| 신뢰성   | 패킷 손실·순서 보장, 재전송   | 보장 안 함, 재전송 없음       |
| 제어 기능 | 흐름/혼잡/오류 제어 제공     | 거의 없음(필요하면 앱에서 처리)   |
| 속도/부하 | 상대적으로 느리고 오버헤드 큼   | 상대적으로 빠르고 오버헤드 작음    |
| 주요 용도 | 웹, 파일전송, 메일, SSH 등 | 게임, 스트리밍, VoIP, 방송 등 |

### REST API

- 서버와 클라이언트가 메시지를 주고받을 때 가장 많이 사용하는 통신 규격 (암묵적인 룰)
- REST (representational state transfer)
- REST: 분산 시스템을 위한 소프트웨어 아키텍처의 한 형태
- RESTful 이란 REST 조건을 만족한다는 뜻
- 요청 주소(URL)과 메서드(GET, POST 등), JSON 규격을 이용하여 API를 정의
- 오늘날 가장 범용적으로 사용
- https://aws.amazon.com/ko/what-is/restful-api/
